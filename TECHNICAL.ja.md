# 技術解説 — SFA3 MAX 16:9 ワイドスクリーンパッチ

> 🌐 [English](TECHNICAL.md) · **日本語** · [Français](TECHNICAL.fr.md) · [Español](TECHNICAL.es.md) · [Português (BR)](TECHNICAL.pt-BR.md)

このパッチをどうリバースエンジニアリングしたか、そして各バイト書き換えが何をするかの
完全な解説です。以下のアドレスは、特記なき限り復号済み EBOOT の **仮想アドレス (VA)** です。

## 表記規則

| 用語 | 値 |
|---|---|
| VA のファイルオフセット | `file = VA + 0x74`（ELF プログラムヘッダが VA 0 → file 0x74 に写像） |
| ランタイムロードベース | `0x08804000` → `runtime = VA + 0x08804000` |
| PSP ネイティブ画面 | 480 × 272（真の 16:9 ハードウェア） |
| 4:3 コンテンツ幅 | 384px → 左右各 48px の余白（センタリングのオフセット） |
| `j`/`jal` のエンコード | `0x08000000 | ((target >> 2) & 0x03FFFFFF)` |

> **再配置の注意。** 手動で注入した `j`/`jal` は PSP ローダーによって **再配置されません**
> （リンク時に存在するジャンプのみ修正されます）。よって注入するジャンプは **ランタイム**
> のターゲット（`VA + 0x08804000`）をエンコードする必要があります。これを誤ると
> 「Bad Execution Address → Jump to 0x00xxxxxx」クラッシュになります（開発中に観測・修正済み）。

## リージョン非依存（パターンスキャン）

EU・US・JP は同じ描画コードを持ちますが **オフセットが異なります**（JP は約 1KB 小さく、
US は EU とバイトの約 30% が異なる）。固定アドレスは当てにならないため、パッチャーは各箇所を
**一意な命令バイトのシグネチャ** で特定し、一致したオフセットにパッチします。各シグネチャは
3 つの EBOOT すべてで期待回数だけ出現することを検証済みです。

## パート 1 — ビューポート拡張

製品版は 480 の中央に 384px 幅のウィンドウを描画し、左右に黒いピラーボックスを表示します。
4 つの書き換えでこれを 480 いっぱいに開きます。

### VP1 — 描画スケーラー幅（`ori a0,zero,0x180` → `0x1E0`）

1 箇所が描画幅として `0x180`（384）を `a0` に読み込みます。**一意な直前のウィンドウ**
`32 00 a5 a7 24 00 a4 a7` で特定。次のワードが `ori a0,zero,0x180`（LE `80 01 04 34`）。
即値の下位バイトを `0x80 → 0xE0` にして `0x1E0` に。

### VP2 — ピラーボックス描画ルーチンの無効化（`jr ra; nop`）

サイドバーを描く関数を、プロローグを `jr ra; nop`（`08 00 e0 03 00 00 00 00`）に置き換えて
無効化。一意な 12 バイトのプロローグシグネチャ `90 ff bd 27  60 00 b0 af  20 00 10 3c` で特定。

### VP 幅 — 8 つの呼び出し箇所（`ori a2/a3,zero,0x180` → `0x1E0`）

8 箇所が GE の幅引数 `0x180` を構築します。リテラルワード `34 06 01 80` は **15 回** 出現
しますが、幅引数なのは 8 つだけ。**8 バイトのウィンドウ** `[ori 0x180][次の命令]` で曖昧さを
解消し、目的の 8 箇所に正確に一致（EU・US で 8 ヒット、他の 7 つとの衝突なし）。下位バイトを
`0x80 → 0xE0`。

### VP カメラ — 11 箇所（`li t0,-192` → `-240`）

カメラ左端テーブルが 11 連続エントリで `-192`（`addiu t0,zero,-0xC0` = `34 08 ff 40`）を
読み込む → `-240`（`34 08 ff 10`）にパッチ。広くなったウィンドウでコンテンツを中央に保つため、
カメラを左へ 48px 多くパンできるようにします。

## パート 2 — 背景タイルレイヤー

### アーキテクチャ（GE + CPU デバッガで判明）

* 背景は **タイル単位で描画されるタイルマップ**：各タイルは GE スルーモードの 1 つの
  `DRAW PRIM` クアッドです。
* 2 つの描画ルーチンがあり、それぞれそのタイルサイズの **すべて** のレイヤーで共有されます：
  * **`FUN_17268`** — 16px タイル（空、雲、岩）。**25 列** 描画。
  * **`FUN_16B28`** — 32px タイル（寺院、滝、前景の草）。**13 列 × 9 行** 描画。
* レイヤーごとの状態構造体（`s0`/`s1`）：`+0x08` ライブ X・`+0x10` 作業用タイルポインタ・
  `+0x18` X ストライド・`+0x20` X 開始・`+0x22` ラップ列・`+0x3E` 行数。
* タイルは **静的な循環バッファ**（64 列）にあり、ステージロード時に一度だけ充填されます。
  スクロールは読み取りウィンドウを動かすだけ。空スロット = `0x00000000`、有効タイルは
  フラグ `0x80000000` を持ちます。
* 共有の頂点エミッタ `FUN_1DD620` が **+48px のセンタリング**（`addiu a3,a3,0x30`）を適用。
  **22 個の呼び出し元**（スプライト、HUD など）があるため **触ってはいけません**。さもないと
  キャラがずれます。

25 列 × 16px ≈ 384px（4:3 コンテンツ）。480 を埋めるには → 約 31 列（6 列追加：左 3 + 右 3）、
中央寄せ。

### 右方向の拡張 — 列数ゲート

| ルーチン | ゲート | 変更前 | 変更後 |
|---|---|---|---|
| `FUN_17268`（16px） | `slti a0,a2,25`（`28 c4 00 19`） | 25 | **31**（`…1F`） |
| `FUN_16B28`（32px） | `slti a3,a2,13`（`28 c7 00 0d`） | 13 | **17**（`…11`） |
| `FUN_16B28`（32px） | `slti a0,a2,13`（`28 c4 00 0d`） | 13 | **17**（`…11`） |

各 32px シグネチャは **2 回** 出現しますが、**`FUN_16B28` の本体内**（行開始フックに最も近い、
0x600 バイト以内）のものだけをパッチ — 無関係な重複は手付かずのままにします。（JP で検証：
遠い重複 0x17ae0 / 0x17d20 は未変更。）

### 左方向の追加（prepend）— コードケーブ

ループは固定の左端から始まり右へ進むため、列数を増やしても **右** 側が伸びるだけです。
既存タイルをピクセル単位で正確に保ったまま **左** に列を追加するため、行開始のセットアップを
フックし、行ごとに：

```
Xstart -= N * stride     ; ウィンドウが N 列早く始まる（480 で再センタリング）
ptr    -= N * 4          ; タイルポインタを N 列戻す（1 列 4 バイト）
```

16px：N=3（Xstart-48, ptr-12）。32px：N=2（Xstart-64, ptr-8）。ポインタと X 開始が同量だけ
動く → 既存タイルは動かず、N 個の新しい実タイルが左の余白に現れます。

**フック A** — `FUN_17268` の行ストア `sh t1,0x08(s0)`（`a6 09 00 08`、一意）→
`j caveA`。ここでの `t1` = ライブ X 開始。
**フック B** — `FUN_16B28` の行ストア `sh t0,0x08(s1)`（`a6 28 00 08`、一意）→
`j caveB`。ここでの `t0` = ライブ X 開始。

> JP のコード生成は行開始部分をわずかに違う形で出力します。パッチャーは一致したクラスタの
> **最後の** ストア、構造的に正しいフック地点を取ります（逆アセンブルで検証：3 リージョン
> すべてでそこの `t1`/`t0` = X 開始）。

**ケーブ A**（7 ワード）は動的に見つけた実行可能パディングの空き（`.text` 内の 56 バイト以上の
ゼロ列、実際には VA `0x1FF1B4`）に：

```
addiu t1,t1,-48      ; Xstart -= 3*16
sh    t1,0x08(s0)    ; ずらされたストアを再現（調整済み）
lw    t4,0x10(s0)    ; 作業用タイルポインタ
addiu t4,t4,-12      ; ptr -= 3 列
sw    t4,0x10(s0)
j     <hookA+8>      ; ランタイム再配置済みの戻り
nop
```

**ケーブ B**（7 ワード、A の直後）：`-64`/`-8`、`s1`/`t0` で同形、`<hookB+8>` へ戻る。

ケーブは **実行可能セグメント内の既存ゼロパディング**（確実に実行できると確認できた唯一の
領域）に置かれます。ファイルサイズの変化はありません。

## 試して却下した手法

* **3 重描画による複製** — 設計上却下（中央からずれる、偽物）。
* **ラップポインタ／ストリーマーのパッチ** — タイルバッファは静的で、ストリーマーはスクロール
  フィールドを更新するだけでタイルは更新しないため、広げても何も得られなかった。
* **クランプ・トゥ・エッジ（空スロットに最後の有効タイルを複製）** — メニューが壊れる。
  「空」は *ステージの端* ではなく *透明* を意味し、レイヤーごとのステージ幅データなしには
  両者を区別できないため。断念。

採用した解決策（prepend + 列数拡張）はフレームを **実際の** タイルで埋め、メニューでも
ゲームプレイでも安定しています。

**wrap 修正（12–15）の理由。** v1.0 のフックはタイルレイヤーの開始を左へ N 列ずらします
（16px は N=3、32px は N=2：`ptr -= N·4`、`Xstart -= N·16`）。これで広がった左余白を埋めます
が、レイヤーごとのラップトリガー（`+0x58` シード → `+0x22`）が **調整されていません**。
レイヤーバッファは 64 列の行優先（ストライド `0x100`）で、ラップ時に巻き戻しはちょうど 1 行分
（`-0x100`）を引きます。開始が N 列早く動いたため、巻き戻しは **前の** 行に N エントリ入った
位置に着地し、継ぎ目の後の列はタイル 1 行上を読み、約 16px **下** に描画されます。修正は
トリガーに `+N` を加え、巻き戻しが行境界にぴったり合うようにします。各修正は 4 ワードのケーブ：
`addiu rt,rt,N ; <元の sh rt,0x22(rs)> ; j site+8 ; nop`。

すべてのパッチは一意なシグネチャで特定されます（EU/US/JP で 1 回の出現を検証。例外は 2 つの
`subu`/count ワードで、これらはコンテキストウィンドウ、または一意な wrap ストアへの近接で
曖昧さを解消）。v1.1 は 6 つのコードケーブ（計 33 ワード）が必要です。

**ケーブの配置（重要）。** 実行イメージ内のすべてのゼロ列が安全とは限りません：一部は
ランタイムのスクラッチ（ファイル上はゼロだが実行時に書き込まれる）で、セグメント末尾の大きな
空きも上書きされます — そこに置いたケーブは上書きされ、フックはゴミへジャンプします
（ブラックスクリーン）。実行時に読み取り専用なのは **セグメント中ほどのコードパディング** だけ
です。そのためアロケータは、昇順スキャンで最初の `>=14` ワードの空き（ゲーム内検証済み v1.0
ビルドが使った領域そのもの）にアンカーし、その局所バンドの **最も大きい空きからのみ** 割り当て
ます（worst-fit、大きいケーブ優先）。結果、6 つのケーブはすべて実証済みの安全なパディング
（全リージョンで `0x1ff1b4`、`0x1fef38`、`0x1feed0`）に着地し、小さい／散在／末尾の空きには
一切触れません。

EU・US・JP すべてでバイトレベル検証済み。JP の自動出力は、ゲーム内で確認済みの手動リファレンス
と同一に描画されます（同じパッチロジック、ケーブも同じ実証済み安全バンド）。
